What does LiveReload do?

LiveReload monitors changes in the file system. As soon as you save a file, it is preprocessed as needed, and the browser is refreshed.

Even cooler, when you change a CSS file or an image, the browser is updated instantly without reloading the page.

Compilers and preprocessors

LiveReload 2 introduces first-class support for CoffeeScript, SASS/SCSS, LESS, Stylus, HAML and Jade, and is shipped with all of them included.

If you want LiveReload to compile your source files, enable the desired items in the project options dialog (the one you get when you double-click a project).

After you add new source files, you might need to go back to the options dialog and set the output directory. LiveReload tries to guess where you want to put the compiled files, however every once in a while its magic dust runs dry and guessing fails.

If compiling through LiveReload is not your thing, keep the compilers disabled on the options page. There are two other modes of operation that LR activates automatically when a compiler is disabled.

First, if you simply want to compile the source files yourself (from the command line or using your own script), LiveReload will step away and ignore the changes to your source files. This mode (total ignorance) is activated when the compiled files are present in the project.

Second, if you are using some kind of server-side middleware to serve the compiled files on the fly, LiveReload simulates a change in a corresponding (non-existent) derived file when you change a source file. FOR EXAMPLE, if you change think.less, LiveReload will tell the browser that think.css has changed. That will cause all CSS files to be reloaded by the browser (unless your middleware actually serves the file as think.css, in which case only that file will be reloaded). This mode is activated if the corresponding compiled files are NOT present in the project.

If that last paragraph is giving you head aches, perhaps you should forget it and just remember that server-side on-the-fly compilation now works out of the box.

What's new in LiveReload?

2.0α10

Fixed Compass support (which has been broken in α8–α9) and a crashing bug on web socket disconnection.

2.0α9

Fixed a web socket protocol bug that caused handshake failures for many users.

2.0α8

Bigger changes:

  1. Completely rewritten browser extensions v2.0. Browser-side code has been split into two parts: livereload.js, which handles all the meaty logic (and can be used stand-alone), and optional browser extensions that merely insert a SCRIPT tag to include livereload.js.
  2. The new livereload.js now supports CSS @import and in-browser LESS compiler, also reconnects automatically.
  3. Server-side @import support for LESS, SASS, Stylus.
  4. Simple support for custom post-processing.
  5. Compiler modes adjusted: new default is a zero-suprise “disabled” mode, the old “ignored” mode has been removed.
  6. For your scripting pleasure, an API based on livereload: URL scheme.

Smaller changes:

  • a rare FSEvents bug that breaks file change monitoring is now automatically detected
  • for those experiencing 'unstyled content after editing CSS' problem, it's now much better (unstyled content flashes instead of staying permanently), and there's a hidden preference to hopefully get rid of it altogether
  • documented another hidden preference to specify additional file extensions to monitor
  • fixed output path computation for newly added compilable files; will now always default to the source file's parent folder when in doubt.
  • fixed multiple selection in compiler's file list to change output directories in bulk
  • added .as and .swf to monitored extensions
  • added node_modules to excluded folders (mainly to avoid compiling LESS/CoffeeScript/whatever inside those folders, even if you enable compilation for the main project)
  • added Slim compiler
  • added support for BBEdit
  • new menu bar icons and smooth animation, kindly provided by Pete Schaffner
  • more usage statistics and privacy policy for those

2.0α7

  1. Fixes memory leaks and CPU utilization issues.
  2. Adds Eco compiler plugin.
  3. Adds monitoring for .ctp extension (CakePHP templates!).
  4. Some minor changes for the compilation error window.

2.0α6

  1. Should be able to connect to the latest Chrome and Firefox (implements WebSockets protocol version 8).
  2. Fixes some bugs and memory leaks.
  3. Includes a new icon artwork.
  4. Full output log can now be displayed for error messages (gears menu » Show Complete Output), displayed automatically when the error message cannot be parsed.
  5. Files and folders starting with a dot (.something.html) are no longer excluded from monitoring (there still is an exclusion list of usual offenders like .git, .svn, CVS and others), per request from ExpressionEngine users.
  6. Any double file extensions are supported now: foo.bar.haml will be compiled into foo.bar.
  7. Monitors more file extensions: phtml, json, yaml, tpl.
  8. LiveReload is now declared an official successor of XRefresh!

2.0α5

  1. Experimental Compass support.
  2. Experimental new compilation error notifications.
  3. Jump to Error works with TextMate, SubEthaEdit, Sublime Text 2, Coda, Espresso or any custom editor if you can write a script.

2.0α4

  1. Correctly handles file names like style.css.sass and script.js.coffee.
  2. Compatible with OS X 10.7 Lion.
  3. Now using a grey background, black no more.

2.0α3

  1. Introduced support for languages that compile into CSS/JS/HTML
  2. Updated the web site to suck less.

2.0α2

  1. Limits monitored extensions to a predefined list. Includes: HTML/CSS/JS and their variants (Less, SASS, HAML, CoffeeScript), images, various PHP extensions, Ruby, Python.
  2. First step of handling Less/SASS/HAML/CoffeeScript. If there is a corresponding css/js file for your less/sass/scss/coffee, the changes to your less/sass/scss/coffee will be ignored (because there is probably an external compiler that will update css/js soon). Otherwise LR proceeds as usual.
  3. The menu bar icon now indicates the state of LiveReload: inactive (no browsers connected, monitoring disabled), active (browser(s) connected, monitoring the file system), change detected (blinks green). You can also drag'n'drop folders onto the menu bar icon.

2.0α1

  1. Native Mac OS X Cocoa-based UI
  2. File system monitoring via native FSEvents API
  3. Does not depend on Ruby or any external libraries
  4. Installs Safari & Chrome extensions (almost) automatically
  5. Monitoring is only activated when a browser is connected — meaning zero resource usage while not in use
